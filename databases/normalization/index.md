# 🗃️ Нормализация баз данных

**Нормализация** — это процесс организации данных в реляционной базе данных. В нормализованной базе минимизировано дублирование данных, с ней проще работать и поддерживать её целостность (избегать аномалий при добавлении, обновлении или удалении данных). 

В процессе нормализации данные разбивают на связанные таблицы, чтобы они занимали меньше места, а структура была гибкой и логичной. Для связи таблиц используются **ключи** (первичные и внешние).

Обычно для правильного проектирования БД достаточно привести её к трем первым нормальным формам (1NF, 2NF и 3NF). Каждая следующая форма включает в себя требования предыдущей.

---

## 1️⃣ Первая нормальная форма (1NF)

**Правило:** В базе данных не должно быть дубликатов и составных данных (списков).
Все значения в столбцах должны быть атомарными (неделимыми), а в одной ячейке не может храниться несколько значений.

### Пример 1: Разделение составных данных
Сложное поле с полным ФИО необходимо разбить на отдельные столбцы, так как иначе будет сложно искать людей только по фамилии.

| Покупатель (Неправильно) | Имя | Отчество | Фамилия |
| :--- | :--- | :--- | :--- |
| Александр Александрович Сушков | Александр | Александрович | Сушков |
| Егор Сергеевич Кузнецов | Егор | Сергеевич | Кузнецов |

### Пример 2: Устранение списков в одной ячейке
Если у одной фирмы несколько моделей, они не должны перечисляться через запятую в одной строке. Каждая модель должна иметь свою собственную строку.

**Было (не 1NF):**

| Фирма | Модели |
| :--- | :--- |
| BMW | M5, X5M, M1 |
| Nissan | GT-R |

**Стало (1NF):**

| Фирма | Модели |
| :--- | :--- |
| BMW | M5 |
| BMW | X5M |
| BMW | M1 |
| Nissan | GT-R |

---

## 2️⃣ Вторая нормальная форма (2NF)

**Правило:** Таблица находится в 1NF, и каждый неключевой атрибут **полностью** зависит от первичного ключа. Не должно быть *частичных зависимостей*.

> **Важный нюанс для собеседований:** Правило 2NF актуально в первую очередь для таблиц с **составным первичным ключом** (когда ключ состоит из двух и более колонок). Если у таблицы простой первичный ключ (одна колонка `ID`) и она уже в 1NF, то она автоматически находится и во 2NF.

### Пример приведения к 2NF
Допустим, у нас есть таблица привязки сотрудников к проектам. Составной первичный ключ здесь: `ID Сотрудника` + `ID Проекта`. 

**Было (не 2NF):**

| ID Сотрудника (PK) | ID Проекта (PK) | Роль в проекте | Имя сотрудника |
| :--- | :--- | :--- | :--- |
| 1 | 100 | Developer | Иван |
| 1 | 101 | QA | Иван |

**Проблема:** Поле `Имя сотрудника` зависит только от `ID Сотрудника`, но никак не зависит от `ID Проекта`. Это частичная зависимость. Имя "Иван" будет дублироваться для каждого нового проекта.

**Стало (нормализация в 2NF):**
Мы выносим информацию о сотрудниках в отдельную сущность, оставляя в сводной таблице только то, что зависит от обеих частей ключа (от связки сотрудника и проекта).

*Таблица сотрудников:*

| ID Сотрудника | Имя сотрудника |
| :--- | :--- |
| 1 | Иван |

*Таблица проектов сотрудника:*

| ID Сотрудника (FK) | ID Проекта | Роль в проекте |
| :--- | :--- | :--- |
| 1 | 100 | Developer |
| 1 | 101 | QA |

---

## 3️⃣ Третья нормальная форма (3NF)

**Правило:** Таблица находится во 2NF, и в ней нет столбцов, которые зависят от других **неключевых** значений. Не должно быть *транзитивных зависимостей*.

Если неключевой атрибут A зависит от неключевого атрибута B, который, в свою очередь, зависит от первичного ключа, то эту связь нужно разорвать.

### Пример приведения к 3NF
Рассмотрим таблицу сотрудников, в которой указаны данные об их подразделении.

**Было (до 3NF):**

| ID (PK) | ФИО | Должность | Подразделение | Описание подразделения |
| :--- | :--- | :--- | :--- | :--- |
| 1 | Иванов И.И. | Программист | Отдел разработки | Разработка и сопровождение... |
| 2 | Сергеев С.С. | Бухгалтер | Бухгалтерия | Ведение бухгалтерского... |

**Проблема:** Поле «Описание подразделения» логически зависит от «Подразделения», а не от ID сотрудника. Возникает транзитивная зависимость: *ID -> Подразделение -> Описание*. Если мы переименуем отдел, нам придется менять это во всех строках сотрудников этого отдела.

**Стало (нормализованная БД в 3NF):**
Для устранения зависимости таблица разбивается на две независимые, связанные по идентификатору (ID) подразделения.

*Таблица сотрудников:*

| ID | ФИО | Должность | Подразделение ID (FK) |
| :--- | :--- | :--- | :--- |
| 1 | Иванов И.И. | Программист | 1 |
| 2 | Сергеев С.С. | Бухгалтер | 2 |

*Таблица подразделений:*

| Подразделение ID (PK) | Название | Описание подразделения |
| :--- | :--- | :--- |
| 1 | Отдел разработки | Разработка и сопровождение приложений и сайтов |
| 2 | Бухгалтерия | Ведение бухгалтерского и налогового учета... |

## 🔄 Денормализация (Denormalization)

Нормализация делает базу идеальной для записи данных (OLTP-системы), так как данные хранятся в одном месте и исключен риск противоречий. Однако при чтении данных из множества нормализованных таблиц базе приходится использовать дорогие операции объединения (`JOIN`).

**Денормализация** — это осознанное и контролируемое нарушение нормальных форм (дублирование данных) ради **увеличения скорости чтения (выполнения SELECT запросов)**. 

Чаще всего денормализация применяется в аналитических базах данных и хранилищах данных (DWH / OLAP), где данные загружаются один раз, а затем быстро читаются для построения тяжелых отчетов и графиков.
