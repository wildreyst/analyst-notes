# 🧩 МК паттерны (Паттерны микросервисов)

Здесь собраны ключевые архитектурные паттерны, применяемые в микросервисных системах: организация взаимодействия, построение UI, развёртывание, мониторинг и стратегии выкатки. Понимание этих шаблонов помогает аналитику грамотно проектировать интеграции и описывать поведение распределённых систем.

---

## 🔄 Паттерны коммуникации микросервисов

### 🌐 API-шлюз (API Gateway)

Шлюз располагается между клиентами и набором микросервисов, предоставляя единую точку входа. В зависимости от задач выделяют разновидности:

- **Gateway Routing** — обратный прокси, перенаправляющий запросы на нужный сервис.
- **Gateway Aggregation** — разветвление запроса на несколько сервисов и агрегация ответов.
- **Gateway Offloading** — выполнение сквозных задач: аутентификация, авторизация, SSL, логирование.

![API Gateway](pics/APIGateway.png)

### 📱 Бэкенды для фронтендов (Backends for Frontends, BFF)

Вариант API Gateway, при котором для каждого типа клиента (Web, Mobile, Desktop) создаётся отдельный шлюз. Позволяет адаптировать API под конкретные нужды клиента, избегая универсального раздутого шлюза. Не рекомендуется, если различия между клиентами минимальны или приложение небольшое.

![BFF](pics/BFF.png)

### 🌀 Сага (Saga)

**Сага** — набор локальных транзакций, каждая из которых обновляет свою БД и публикует событие для запуска следующего шага. При неудаче выполняются компенсирующие транзакции, отменяющие предыдущие изменения.

**Способы координации саги:**

- **Хореография (Choreography)** — сервисы обмениваются событиями напрямую, без центрального управляющего. Каждый участник сам знает, на какие события реагировать.
- **Оркестрация (Orchestration)** — центральный оркестратор явно вызывает сервисы в нужном порядке и обрабатывает сбои.

### ⚖️ Two-Phase Commit (2PC)

**2PC** — протокол распределённых транзакций, гарантирующий консистентность: либо все участники подтверждают операцию, либо ни один. Состоит из фазы подготовки (координатор опрашивает готовность) и фазы подтверждения (commit/rollback). Требует блокировки ресурсов на всё время транзакции, что может быть неприемлемо для высоконагруженных систем.

В отличие от саги, где откат реализуется компенсирующими действиями, 2PC использует блокировки и строгую атомарность.

![2PC пример 1](pics/2PCex1.png)
![2PC пример 2](pics/2PCex2.png)

### 📬 Транзакционный Outbox (Transactional Outbox)

Гарантирует надёжную доставку сообщений: данные о событии сохраняются в отдельной таблице (outbox) в рамках той же транзакции, что и бизнес-операция. Затем отдельный процесс публикует сообщения в брокер. Так достигается атомарность: либо фиксируются и данные, и сообщение, либо ничего.

## 🖥️ Паттерны построения пользовательского интерфейса

### 🧩 Сборка UI на стороне клиента (Client-Side UI Composition)

Каждый экран разбивается на фрагменты, за которые отвечают разные микросервисы. Браузер собирает итоговую страницу из независимых мини-приложений (например, на Angular, React). Обновление фрагментов происходит без перезагрузки всей страницы (SPA).

![Client-Side UI Composition](pics/ClientSideUI.png)

### 📄 Сборка фрагментов страниц на стороне сервера (Server-Side Page Fragment Composition)

Сборка контента происходит на серверной стороне — обычно отдельным слоем (Nginx, Varnish, CDN), который агрегирует фрагменты от разных сервисов и возвращает клиенту готовую страницу. Ускоряет загрузку, снижает нагрузку на клиент.

![Server-Side Page Fragment Composition](pics/ServerSidePageFragmentComposition.png)

## 🚀 Паттерны развертывания микросервисов

### 🏠 Экземпляр сервиса на хост (Service Instance Per Host)

Каждый экземпляр сервиса разворачивается на собственном хосте (физическом или виртуальном). Обеспечивает изоляцию, предотвращает конфликты версий и упрощает повторное развёртывание. На практике чаще применяют варианты:

- **Экземпляр на виртуальную машину (Service Instance Per VM)**
- **Экземпляр на контейнер (Service Instance Per Container)** — например, Docker-контейнеры.

![Service Instance Per Host](pics/ServiceInstancePerHost.png)

### 🔵🟢 Сине-зелёное развертывание (Blue-Green Deployment)

Поддерживаются две идентичные production-среды: синяя (текущая) и зелёная (новая). После развёртывания и тестирования новой версии трафик переключается на зелёную среду. В случае проблем можно быстро вернуться к синей. Минимизирует время простоя и риски.

![Blue-Green Deployment](pics/BlueGreenDeployment.png)

## 📊 Паттерны мониторинга микросервисов

### 📜 Агрегация логов (Log Aggregation)

Централизованная система собирает логи со всех экземпляров сервисов, предоставляя единую точку для поиска, анализа и настройки алертов. Реализуется, например, через ELK-стек.

![Log Aggregation](pics/LogAggregation.png)

### 🔗 Распределённая трассировка (Distributed Tracing)

Каждому входящему запросу присваивается уникальный идентификатор (TraceId), который передаётся по всей цепочке вызовов и записывается в логи. Это позволяет отследить путь запроса и точно локализовать ошибки в распределённой системе.

![Distributed Tracing](pics/DistributedTracing.png)

## 🚢 Дополнительные шаблоны развёртывания приложений

### 🐤 Канареечный релиз (Canary Release)

Новая версия выкатывается на небольшую группу пользователей. После подтверждения стабильности и отсутствия проблем она распространяется на всех. Позволяет выявить дефекты до массового воздействия.

### 🔵🟢 Сине-зелёное развёртывание (Blue/Green)

Уже описан выше: две среды, быстрое переключение трафика, лёгкий откат.

### 🎚️ Фича-тогглинг (Feature Toggles)

Возможность включать/выключать отдельные функции во время выполнения, не разворачивая новую версию целиком. Разделяет выпуск кода и активацию функциональности.

### 🆎 A/B-тестирование

Сравнение двух вариантов приложения на реальных пользователях: части аудитории показывается версия A, другой — версия B. Статистически определяется более эффективная реализация.

### 🌑 Тёмный запуск (Dark Launch)

Новая функция скрытно внедряется для избранной группы пользователей, которые не подозревают об этом. Собираются обратная связь и метрики производительности без явного анонсирования.
