# ⚙️ Use Case (Варианты использования)

**Use Case (Вариант использования или прецедент)** — это структурированный метод описания функциональных требований к системе. Он представляет собой текстовое описание того, как пользователи (или другие системы) взаимодействуют с разрабатываемой системой шаг за шагом для достижения определенной цели.

### Зачем нужны Use Case?
* **Единое понимание:** Помогают перевести бизнес-идеи на язык взаимодействия, понятный как заказчикам, так и команде разработки.
* **Определение границ:** Четко показывают, что должна делать система, а что остается зоной ответственности пользователя или внешних сервисов.
* **Основа для тестирования и UI:** Пошаговые сценарии — это идеальная база для написания тест-кейсов (QA) и проектирования пользовательских интерфейсов (UX/UI).
* **Управление логикой:** Позволяют заранее продумать не только «идеальный» путь пользователя, но и то, как система должна реагировать на ошибки и нестандартные ситуации.

---

## 📝 Структура описания Use Case

Ниже представлен стандартный шаблон для документирования варианта использования.

| Атрибут | Описание |
| :--- | :--- |
| **Имя сценария** | Идентификатор + название варианта использования (например, *UC-01 Оформление заказа*). |
| **Цель** | Кратко описывает то, чего пользователь намеревается достигнуть с этим сценарием. |
| **Действующие лица (Actors)** | Кто-то или что-то вне системы, влияющий на нее или находящийся под её влиянием (человек, устройство, другая система или подсистема). |
| **Триггеры** | Событие или условие, которое заставляет актера приступить к выполнению Use Case. |
| **Предварительные условия (Preconditions)** | Определяет, что должно присутствовать или быть выполнено до выполнения Use Case в системе. |
| **Основной сценарий (Main flow)** | Последовательность действий, которые выполняет участник для успешного достижения цели. |
| **Альтернативный сценарий (Alternate flow)** | Может привести к успешному выполнению задания и удовлетворяет выходным условиям. |
| **Исключающий сценарий (Exceptional flows)** | Описывает ожидаемое ошибочное условие, которое может сложиться во время выполнения Use Case, и как его обрабатывать. |
| **Выходные условия (Postconditions)** | Описывают состояние системы после успешного выполнения Use Case, т.е. какой результат после выполнения Use Case получит действующее лицо. |

## ⚖️ Преимущества и недостатки

### ✅ Преимущества:
* **Полнота описания:** Детально прорабатывают логику, не оставляя «слепых зон» во взаимодействии.
* **Фокус на взаимодействии:** Описывают систему с точки зрения приносимой ценности и реального поведения акторов.
* **Удобны для сложных систем:** Отлично подходят для систем со сложной бизнес-логикой, где много ветвлений, статусов и интеграций.

### ❌ Недостатки:
* **Сложность и объем:** Создание и поддержка таких документов в актуальном состоянии требует больших затрат времени.
* **Избыточная детализация:** Могут быть слишком подробными, что затрудняет их использование на более высоком уровне абстракции (когда бизнесу нужно просто понять общую концепцию).
* **Нет описания нефункциональных требований:** В этом формате невозможно корректно описать требования к производительности, безопасности, нагрузке.
