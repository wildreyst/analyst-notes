# 🏃 Диаграмма действий (Activity Diagram)

**Диаграмма действий (Activity Diagram)** — это поведенческая диаграмма UML, которая показывает последовательность переходов от одного действия к другому, варианты ветвлений и их результаты. 

По своей сути она очень похожа на классическую блок-схему (Flowchart), но обладает расширенными возможностями: например, умеет наглядно показывать параллельные процессы. В системном анализе она часто используется как более техническая альтернатива нотации BPMN.

## 🧱 Основные элементы (Нотация)

Для чтения и построения Activity Diagram используется стандартизированный набор графических примитивов:

| Элемент | Название в UML | Описание |
| :--- | :--- | :--- |
| **⚫ (Сплошной черный круг)** | **Начальный узел** *(Initial Node)* | Точка входа. Показывает, с чего начинается процесс. На диаграмме может быть только одно начальное состояние (в большинстве случаев). |
| **🟦 (Прямоугольник со скругленными углами)** | **Действие** *(Action / Activity)* | Единичный шаг алгоритма. Обозначает конкретное действие, которое выполняет система или пользователь (например, «Проверить баланс», «Отправить email»). |
| **🔶 (Ромб)** | **Ветвление / Слияние** *(Decision / Merge)* | Точка принятия решения. Из ромба выходят несколько стрелок с условиями (например, `[Баланс > 0]` и `[Баланс < 0]`). Также ромб используется для обратного слияния альтернативных веток в один поток. |
| **➖ (Жирная черная линия)** | **Разделение / Соединение** *(Fork / Join)* | **Главная фишка диаграммы.** Используется для показа *параллельных* процессов. Одна линия делится на несколько (Fork), чтобы показать задачи, выполняемые одновременно. Затем они снова сходятся в одну черту (Join), означая, что процесс пойдет дальше только когда завершатся все параллельные ветки. |
| **🔘 (Черный круг в белой рамке)** | **Конечный узел** *(Final Node)* | «Бычий глаз». Означает полное и успешное завершение всего процесса. |
| **🛑 (Крестик в круге)** | **Узел завершения потока** *(Flow Final Node)* | Завершает только одну конкретную параллельную ветку, не останавливая работу всего процесса в целом. |

## 🏊‍♂️ Дорожки (Swimlanes / Partitions)

Как и в BPMN, в диаграмме активностей можно использовать **дорожки (Swimlanes)**. 
Они делят полотно диаграммы на вертикальные или горизонтальные зоны. Каждая зона представляет отдельного участника процесса (Actor) или компонент системы (например: *«Пользователь»*, *«Фронтенд»*, *«Платежный шлюз»*, *«База данных»*). 

Действия помещаются в ту дорожку, чья роль ответственна за выполнение этого шага. Это позволяет мгновенно понять, кто именно и на каком этапе выполняет работу.

## 🆚 Activity Diagram против BPMN

Аналитику важно понимать, когда лучше использовать UML Activity Diagram, а когда — BPMN (Business Process Model and Notation).

**Почему выбирают Activity Diagram:**
1.  **Более техничный фокус:** Идеально подходит для описания внутренних алгоритмов программы (например, логики работы конкретной функции на уровне кода или базы данных).
2.  **Простота:** Нотация Activity Diagram содержит гораздо меньше элементов, чем BPMN. Её проще нарисовать и легче объяснить разработчикам (которые лучше знают UML, чем бизнес-нотации).
3.  **Детализация Use Cases:** Часто используется для пошаговой детализации одного конкретного сценария использования (Use Case).

**Почему выбирают BPMN:**
1.  **Бизнес-фокус:** Разработан специально для бизнес-аналитиков и менеджеров. Отлично описывает бизнес-процессы на уровне всей компании.
2.  **Событийность:** В BPMN есть огромный набор специфичных событий (события времени, сообщения, ошибки, эскалации), которых не хватает в базовом UML.
3.  **Ориентированность на людей:** BPMN лучше показывает документооборот и ручное взаимодействие людей (Human Tasks) между отделами.
