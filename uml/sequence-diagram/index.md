# ⏱️ Диаграмма последовательности (Sequence Diagram)

**Sequence diagram (Диаграмма последовательности)** — это поведенческая UML-диаграмма, которая наглядно представляет совокупность разных элементов модели системы, изображая то, как, в каком порядке и в течение какого времени они взаимодействуют друг с другом. 

Она незаменима для интеграционных аналитиков, так как позволяет детально спроектировать межсистемное взаимодействие (API, очереди сообщений) в хронологическом порядке (сверху вниз).

## 🧱 Комбинированные фрагменты (Управляющие конструкции)

Для реализации логических разветвлений, циклов и условий на диаграмме используются специальные рамки — фрагменты:

*   **`alt` (Alternative):** Несколько альтернативных фрагментов. Работает как ветвление `if-else`. Выполняется только тот фрагмент, логическое условие которого истинно.
*   **`opt` (Optional):** Необязательный фрагмент. Выполняется, только если условие истинно. Эквивалентно оператору `if` без ветки `else`.
*   **`par` (Parallel):** Параллельный фрагмент. Все процессы, находящиеся внутри этой рамки, выполняются СУБД или сервисами одновременно.
*   **`loop` (Цикл):** Повторяющийся фрагмент. Блок кода внутри рамки выполняется несколько раз, пока условие (защита) итерации остается верным.

## 🛠 Кейс 1: Покупка акций и начисление подарка в МП

### 1. PlantUML код диаграммы
Ниже представлен полностью спроектированный и завершенный код сквозного процесса, разбитый на 5 требуемых бизнес-подпроцессов:

```plantuml
@startuml
actor "Клиент" as Client
participant "Мобильное\nприложение (МП)" as Mob
participant "Auth-сервис" as Auth
participant "Asset-сервис" as Asset
database "БД Asset" as DB
participant "Gift-сервис" as Gift
participant "Payment-сервис" as Payment
participant "Push-сервис" as Push

== 1. Авторизация пользователя ==
activate Client
Client -> Mob ++ : Открытие приложения, ввод учетных данных
Mob -> Auth ++ : POST /api/v1/auth (Запрос токена)
Auth -->> Mob -- : 200 OK (JWT Токен доступа)

alt#Gold Успешная авторизация
    Mob -->> Client : Отображение главного экрана (Витрина акций)
else Ошибка авторизации (Неверный пароль / Истек токен)
    Mob -->> Client : Экран ошибки / Форма повторного ввода
    break
        note over Mob: Прерывание сценария
    end
end

== 2. Покупка акций в приложении ==
Client -> Mob : Нажатие кнопки "Купить акции" на N рублей
Mob -> Asset ++ : POST /api/v1/orders/buy (Данные сделки)
Asset -> DB ++ : Сохранение информации о транзакции
DB -->> Asset -- : Транзакция зафиксирована
Asset -->> Mob : 201 Created (Акцепт покупки)
Mob -->> Client -- : Экран "Акции успешно куплены!"

== 3. Процесс анализа покупок (Маркетинг) ==
note over Asset, Gift: Асинхронное уведомление о событии покупки
Asset -> Gift : Event: OrderCreated (Публикация в брокер сообщений)
deactivate Asset

activate Gift
note over Gift: Анализ участия клиента в акции \nи проверка критериев рекламной кампании

== 4. Процесс начисления "подарка" пользователю ==
opt Клиент проходит по критериям рекламной кампании
    Gift -> Payment ++ : POST /api/v1/bonus/credit (Начисление средств)
    note over Payment: Проведение операции зачисления \nподарочных средств на счет клиента
    Payment -->> Gift -- : 200 OK (Подарок успешно зачислен)
    
    == 5. Уведомление клиенту (Push) ==
    Gift -> Push ++ : Event: GiftAwarded (Команда на отправку уведомления)
    deactivate Gift
    Push -> Mob ++ : Отправка Push-уведомления (Firebase / APNS)
    Mob -> Client : Всплывающий Пуш: "Вам начислен подарок!"
    Mob -->> Push -- : Подтверждение доставки пуша
    deactivate Push
end
@enduml
```

### 2. Виды интеграций между компонентами
*   **Синхронное взаимодействие (REST API over HTTP):**
    *   `МП -> Auth-сервис` и `МП -> Asset-сервис`. Требует мгновенного ответа (ожидание пользователя перед экраном).
    *   `Gift-сервис -> Payment-сервис`. Синхронный вызов команды для гарантированного проведения финансовой транзакции начисления.
*   **Асинхронное взаимодействие (Event-Driven через Message Broker — Kafka/RabbitMQ):**
    *   `Asset-сервис -> Gift-сервис`. Публикация события `OrderCreated`. Покупка акций не должна зависеть от доступности маркетингового сервиса подарков.
    *   `Gift-сервис -> Push-сервис`. Отправка события `GiftAwarded` на нотификацию, выполняемая в фоновом режиме.

### 3. Структура данных (Пример JSON-сообщения при покупке акций)
**Запрос от МП в `Asset-сервис` (`POST /api/v1/orders/buy`):**
```json
{
  "clientId": "usr_998241ad",
  "assetTicker": "GAZP",
  "currency": "RUB",
  "orderType": "MARKET",
  "totalAmount": 15000.00,
  "quantity": 110
}
```

## 🛠 Кейс 2: Оформление кредитной карты через сайт (В2В / Логика)

**Бизнес-задача:** Разработать фичу выдачи кредитной карты через сайт. 
**Системы-участники:** Сайт, СУЗ (Система учета заявок), Скоринг, Печать, Логистика.

### 1. PlantUML код диаграммы

```plantuml
@startuml
participant "Сайт" as Web
participant "СУЗ" as SUZ
participant "Скоринг" as Scoring
participant "Печать" as Print
participant "Логистика" as Logistics

== Этап 1. Регистрация заявки ==
Web -> SUZ ++ : POST /api/v1/applications (Данные анкеты клиента)
SUZ -> SUZ : Создание заявки в статусе "Новая"
SUZ -->> Web : 201 Created (ID_Заявки: 555)

== Этап 2. Проверка благонадежности ==
SUZ -> Scoring ++ : POST /api/v1/scoring/check (Данные для проверки)
note over Scoring: Анализ кредитной истории \nи расчет надежности клиента
Scoring -->> SUZ -- : 200 OK (Результат: Одобрено, Лимит: 100к)

alt#Gold Результат проверки: Одобрено
    SUZ -> SUZ : Смена статуса заявки на "Одобрено"
    
    == Этап 3. Выпуск карты ==
    SUZ -> Print ++ : POST /api/v1/cards/issue (ID_Заявки, Данные эмбоссирования)
    note over Print: Физическое изготовление \nи чипирование пластиковой карты
    Print -->> SUZ -- : 200 OK (Карта выпущена, Номер: XXXX)
    
    == Этап 4. Доставка карты ==
    SUZ -> Logistics ++ : POST /api/v1/deliveries (Данные клиента, ID_Карты)
    Logistics -> Logistics : Назначение курьера \nи построение маршрута
    Logistics -->> SUZ -- : 200 OK (Заказ на доставку создан)
    
    SUZ -> SUZ : Смена статуса заявки на "Передано в доставку"

else Результат проверки: Отказ
    SUZ -> SUZ : Смена статуса заявки на "Отклонено"
    note over SUZ: Процесс завершается отказом
end
@enduml
```
