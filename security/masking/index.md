# 🎭 Маскирование данных

**Маскирование данных** — это процесс замены конфиденциальной или чувствительной информации на фиктивные, но реалистичные данные. 

Этот процесс используется для защиты данных в сценариях, где реальные данные не требуются или опасны (тестирование, разработка, анализ, машинное обучение). В банковской сфере маскирование критически важно для защиты номеров счетов, личных данных и транзакций клиентов.

---

## 💡 Шпаргалка для собеседования

Чтобы уверенно отвечать на вопросы интервьюера, достаточно понимать три базовые вещи:

**1. В чем главная разница между шифрованием и маскированием?**
*   **Шифрование — обратимо.** Данные превращаются в нечитаемый набор символов (`#$d8!k`), но при наличии ключа их можно вернуть в исходный вид.
*   **Маскирование — необратимо.** Данные заменяются на другие, структурно правильные (Иван -> Петр). Оригинал уничтожается, восстановить его из замаскированного значения невозможно.

**2. Зачем маскировать данные, если можно просто сгенерировать фейковую базу для тестов?**
Сгенерированные "с нуля" фейковые данные обычно слишком идеальные. Разработчикам и тестировщикам нужна реальная статистика, реальные объемы и реальные ошибки пользователей (нестандартные форматы), чтобы отлавливать сложные баги. Маскирование дает 100% реалистичность без нарушения законов о персональных данных.

**3. Когда применять статическое (SDM), а когда динамическое (DDM) маскирование?**
*   **Статическое (SDM):** Нужно отдать базу данных разработчикам или внешнему подрядчику. Делается копия продакшена, прогоняется через скрипт-маскировщик и физически передается в тестовый контур.
*   **Динамическое (DDM):** Сотрудник техподдержки смотрит в "живую" базу. Сама база не меняется, но система "на лету" подменяет данные перед выводом на экран (показывает `5432 **** **** 1234` вместо полного номера карты).

## 🛠 Задачи системного аналитика при маскировании

При внедрении процессов маскирования аналитик выполняет следующие шаги:

1. **Определение чувствительных данных:** Идентификация атрибутов, требующих защиты (ФИО, номера счетов, ИНН, адреса, телефоны).
2. **Выбор метода маскирования:** (Статическое, Динамическое, Детерминированное — замена всегда на одно и то же значение для сохранения связей в БД, Недетерминированное — случайная замена).
3. **Разработка правил маскирования:** Написание алгоритмов замены (например, замена всех имен на случайные из справочника или генерация случайных 10 цифр для счета).
4. **Интеграция в процессы:** Настройка инструментов извлечения баз данных для автоматического маскирования.
5. **Тестирование и валидация:** Проверка того, что замаскированные данные выглядят реалистично, проходят валидацию форматов и не ломают бизнес-логику системы.
6. **Документирование:** Фиксация алгоритмов и правил для команды разработки.

## 📊 Пример маскирования данных

Предположим, у нас есть таблица с личными данными клиентов банка. Мы хотим замаскировать их для передачи в отдел тестирования.

| Атрибут | Исходные данные | Маскированные данные | Правило маскирования (Алгоритм) |
| :--- | :--- | :--- | :--- |
| **Имя** | Иван Иванов | Алексей Смирнов | Подмена на случайное значение из готового справочника имен. |
| **Счет** | `1234567890` | `9876543210` | Рандомизация с сохранением исходного формата (строго 10 цифр). |
| **Адрес** | ул. Ленина, д. 10 | ул. Садовая, д. 5 | Замена на случайный, но синтаксически корректный адрес. |
