# 🔄 WebSocket

**WebSocket** — протокол полнодуплексной связи поверх TCP, предназначенный для обмена сообщениями в реальном времени между клиентом и сервером. В отличие от REST, где клиент постоянно опрашивает сервер, WebSocket устанавливает постоянное соединение, через которое сервер может самостоятельно отправлять данные клиенту. Это делает его незаменимым для интерактивных приложений: чатов, онлайн-игр, биржевых котировок и live-уведомлений.

---

## ⚙️ Как работает WebSocket

1. **Установка соединения (Handshake)**  
   Клиент отправляет HTTP-запрос с заголовками `Upgrade: websocket` и `Connection: Upgrade`. Сервер, поддерживающий WebSocket, отвечает `101 Switching Protocols`, и TCP-соединение переходит в режим постоянного двустороннего канала.

2. **Схема адресации**  
   - `ws://` — обычное соединение (аналог HTTP)  
   - `wss://` — защищённое соединение (аналог HTTPS)

3. **Обмен данными**  
   После рукопожатия клиент и сервер могут отправлять друг другу **фреймы** (текстовые или бинарные) в любой момент без дополнительных HTTP-заголовков. Формат данных чаще всего JSON, но может быть любым.

4. **Состояния готовности**  
   - `CONNECTING` — соединение устанавливается  
   - `OPEN` — можно передавать данные  
   - `CLOSING` — инициировано закрытие  
   - `CLOSED` — соединение разорвано

## 📊 Сравнение с REST и Webhook

| Критерий               | REST (HTTP)              | WebSocket                     | Webhook                       |
|------------------------|--------------------------|-------------------------------|-------------------------------|
| **Направление связи**  | Клиент → Сервер          | Двусторонняя (↔)              | Сервер → Клиент (одностор.)   |
| **Инициатор**          | Клиент                   | Обе стороны                   | Сервер (по событию)           |
| **Соединение**         | Открывается на 1 запрос  | Постоянное                    | Разовое (HTTP callback)       |
| **Протокол**           | HTTP/1.1, HTTP/2         | WebSocket (поверх TCP)        | HTTP (POST на зарегистр. URL) |
| **Задержка**           | Высокая (polling)        | Минимальная (real-time)       | Средняя (зависит от retry)    |
| **Типичные сценарии**  | CRUD-операции, API       | Чаты, игры, биржи, IoT        | Уведомления о событиях, CI/CD |

## ✅ Преимущества и недостатки

**Плюсы:**
- Постоянное соединение без повторных рукопожатий (низкие накладные расходы)
- Мгновенная двусторонняя передача данных (real-time)
- Поддержка бинарных фреймов и текста (JSON, MessagePack)
- Меньшая нагрузка на сервер, чем при частых polling-запросах

**Минусы:**
- Требует постоянного TCP-соединения (ресурсы сервера)
- Прокси-серверы и старые брандмауэры могут блокировать WebSocket
- Нет встроенной поддержки кэширования или роутинга, как в REST
- Необходимость самостоятельно реализовывать механизм переподключения при обрыве

---

*WebSocket идеален, когда нужно быстро передавать обновления в обе стороны. Для уведомлений в одну сторону часто достаточно более простого механизма — [Webhook](/integrations/webhook/).*

## 📦 Пример использования (JavaScript)

**Клиентская сторона:**

```javascript
// Подключение к серверу WebSocket
const socket = new WebSocket('wss://example.com/socket');

// Слушаем открытие соединения
socket.addEventListener('open', (event) => {
    socket.send('Привет, сервер!');
});

// Получаем сообщения от сервера
socket.addEventListener('message', (event) => {
    console.log('Сообщение от сервера:', event.data);
});

// Обрабатываем закрытие
socket.addEventListener('close', (event) => {
    console.log('Соединение закрыто', event.code, event.reason);
});
