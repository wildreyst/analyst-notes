# 🔔 Webhook

**Webhook** — механизм, позволяющий одному приложению автоматически уведомлять другое о событиях в реальном времени с помощью HTTP-запроса. В отличие от REST, где клиент постоянно опрашивает сервер на предмет изменений, webhook сам отправляет данные при наступлении условия, выступая в роли «обратного вызова».

Webhook-уведомления широко применяются в CI/CD-пайплайнах, платежных системах, мессенджерах, CRM и любых сценариях, где важно мгновенно реагировать на изменения.

---

## ⚙️ Как работает Webhook

1. **Регистрация подписчика**  
   Приложение-получатель заранее предоставляет отправителю URL-адрес (вебхук), на который будут приходить уведомления.

2. **Событие на стороне отправителя**  
   Когда в системе-источнике происходит заданное событие (создан платёж, запущен билд, изменена запись), она формирует HTTP-запрос.

3. **Отправка данных**  
   Отправитель выполняет `POST`-запрос на зарегистрированный URL. Тело запроса обычно содержит JSON с информацией о событии: например, данные о коммите в репозитории или детали платежа.

4. **Обработка получателем**  
   Приложение-получатель обрабатывает входящий запрос, выполняет необходимую бизнес-логику и возвращает код `200 OK` для подтверждения успешного приёма.

## 📊 Сравнение с REST и WebSocket

| Критерий               | REST (HTTP)              | Webhook                       | WebSocket                     |
|------------------------|--------------------------|-------------------------------|-------------------------------|
| **Инициатор**          | Клиент                   | Сервер (по событию)           | Обе стороны                   |
| **Направление**        | Клиент → Сервер          | Сервер → Клиент               | Двустороннее (↔)              |
| **Соединение**         | Разовое (на 1 запрос)    | Разовое (HTTP callback)       | Постоянное                    |
| **Протокол**           | HTTP/1.1, HTTP/2         | HTTP (обычно POST)            | WebSocket (поверх TCP)        |
| **Задержка**           | Зависит от polling       | Минимальная (сразу по событию)| Минимальная (real-time)       |
| **Типичные сценарии**  | CRUD, API                | Уведомления, CI/CD, платежи   | Чаты, игры, биржи, IoT        |

## 🔐 Безопасность Webhook

Поскольку URL вебхука общедоступен, важно убедиться, что запрос пришёл именно от ожидаемого отправителя.

- **Проверка подписи (HMAC)**  
  Отправитель подписывает тело запроса секретным ключом и передаёт подпись в заголовке (например, `X-Hub-Signature-256`). Получатель сверяет подпись.

- **Использование HTTPS**  
  Всегда применять `https://` для шифрования трафика.

- **Валидация IP-адресов**  
  Принимать запросы только с известных IP-адресов отправителя (если провайдер предоставляет список).

- **Ограничение повторных попыток (retry policy)**  
  Корректно обрабатывать повторные доставки одного и того же события (идемпотентность).

## ✅ Преимущества и недостатки

**Плюсы:**
- Мгновенное уведомление о событиях (без polling)
- Простота реализации на стороне получателя (обычный HTTP-эндпоинт)
- Низкая нагрузка на сеть: запрос только при реальном событии
- Широкая поддержка во множестве сервисов (GitHub, Stripe, Telegram, Slack)

**Минусы:**
- Требует публично доступного URL (сложно для локальной разработки)
- Отправитель может не гарантировать порядок или однократность доставки
- Сложность отладки: нужен инструментарий для перехвата входящих запросов
- Безопасность полностью лежит на получателе (проверка подписей, валидация)

---

## 🧪 Инструменты для локального тестирования

- **ngrok** — создаёт публичный туннель к локальному серверу, позволяя принимать webhook на машине разработчика.
- **webhook.site** — генерирует временный URL для приёма и просмотра входящих запросов без написания кода.

---

*Webhook — идеальный выбор, когда нужно реагировать на события в реальном времени без постоянного опроса. Для двусторонней связи в реальном времени лучше подходит WebSocket*
